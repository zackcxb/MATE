# 入口文档降噪与清理

## 目标

确保入口文档只保留“下一位 agent 立即需要知道”的高信噪比信息，不把历史细节、被推翻决策或重复索引继续堆在入口层。

## 触发条件

1. 更新 `docs/project-context.md`。
2. 重写 handoff prompt。
3. 阶段设计/impl pre-plan 被 reset、retarget 或重新定义问题。
4. 一个文档在连续几轮后明显变长，但当前阶段没有相应增加复杂度。

## 核心判据

对每条信息标注以下三项：

1. `实时性`：是否仍然描述当前阶段，而不是上一个阶段。
2. `有效性`：是否仍然帮助下一步决策、执行或 handoff。
3. `间接性`：是否更适合改为一句摘要 + 指向原文链接。

## 执行流程

1. 锁定文档角色。
   - 入口文档的职责是“快速定向”，不是“完整历史归档”。
2. 逐段扫描并标注处理动作：
   - `keep`：保留，当前仍必要。
   - `trim`：保留核心句，删掉细节。
   - `move`：迁移到 `docs/retros/`、`docs/plans/`、`docs/ref/`。
   - `link-only`：只保留一句结论和目标文档路径。
   - `delete`：已失效或纯噪声。
3. 优先删除四类高噪声内容：
   - 被推翻后的旧结论与旧 TODO
   - 详细验证日志、环境参数、命令、产物清单
   - 与邻接入口文档重复的索引或出口条件
   - “也许以后有用”的历史段落
4. 重写文档首屏。
   - 前 30-50 行应让新 agent 明白当前阶段、冻结事实、开放问题、下一步。
5. 交叉复核相邻入口文档。
   - 若 `project-context` 被重定向，检查 handoff prompt / design reset / pre-plan 是否还残留旧叙事。

## 输出

1. 精简后的入口文档。
2. 一个简短审查结论：
   - 删了什么
   - 为什么删
   - 细节现在放在哪

## 完成标准

1. 文档首屏即可回答“当前阶段是什么、当前问题是什么、下一步是什么”。
2. 详细证据留在对应 retro / plan / ref，而不是继续堆在入口文档。
3. 文档中不存在已被推翻但未删除的结论、待办或阶段叙事。
4. 相邻入口文档之间没有明显重复的大段结构或出口条件。

## 常见误区

1. “先追加新结论，旧内容先不删。”
   - 结果：入口文档同时承载互相冲突的阶段叙事。
2. “验证证据很重要，所以放在入口文档最安全。”
   - 结果：入口文档退化成 retro。
3. “索引越全越好。”
   - 结果：真正重要的当前文档反而被淹没。
4. “这是历史事实，不能删。”
   - 结果：把事实层和入口层混在一起。历史事实应保存在正确的归档文档中。
