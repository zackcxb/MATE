# 灵活代码变更解释器

## 目标

按用户指定范围解释代码变更，回答“改了什么、为什么改、有什么影响”，并提供可复现验证命令。

## 触发条件

1. 用户明确要求解释代码变更。
2. PR 前统一解释。
3. 里程碑收口需要变更说明。

固定触发词（可选）：`解释代码变更`、`复盘代码修改`。

> 注：触发条件 2（PR 前）和 3（里程碑收口）同时触发 Mode B；其他情况默认 Mode A。

## 模式选择

| 模式 | 适用场景 | 粒度 |
|------|---------|------|
| **Mode A（默认）** | 开发中、新增模块、阶段性回顾 | 模块职责 + 模块间关系 |
| **Mode B** | PR 准备、里程碑收口、代码审查 | 文件 + 函数级变更清单 |

选择规则：
- 默认 Mode A。
- 用户触发词含"PR前"、"里程碑收口"或"代码审查"时切换 Mode B。

## 执行步骤

1. **锁定范围**：优先响应用户的具体要求；未指定时默认最近 1 次 commit。
2. **收集证据**：从对话历史、 git diff、文件改动、测试结果中抽取证据；无证据不做推断。
3. **使用独立 explainer subagent**：解释任务统一走 dedicated subagent，主线程只回传结构化结果与摘要。
4. **结构化解释**：按所选模式输出。

   **Mode A 输出结构：**
   - **变更范围**：覆盖的 commit / 时间窗口
   - **新增模块概述**：每个新增模块一行，格式：`模块名：一句话职责 + 在系统中的角色/位置`
   - **模块间关系**：用简短调用链或文字说明模块如何协作（不列函数细节）
   - **当前边界**：未覆盖的集成点或已知限制

   **Mode B 输出结构：**
   - **变更范围**：明确本次解释覆盖的对话轮次/文件/时间/Git提交
   - **改了什么**：分文件列出核心代码变更（新增/删除/修改的函数、变量、逻辑）
   - **为什么改**：结合需求背景、对话历史或代码逻辑，说明改动原因
   - **有什么影响**：说明改动的影响范围、边界条件、是否需要同步修改其他内容

## 输出要求

- 若无法明确改动原因，直接说明“需结合更多需求背景确认”，不凭空猜测
- Mode A 控制在 300 字以内，优先说清楚模块职责和关系，不列函数签名与实现细节
- Mode B 若变更较多，优先解释核心文件的核心改动，次要改动可简要带过，控制在 500 字以内
- 输出语言简洁清晰，避免过度技术化堆砌

## 红线

1. 禁止越过用户指定范围扩展解释。
2. 禁止在无证据情况下给出原因结论。
3. Mode A 禁止输出函数签名、参数列表、TDD 驱动证据等实现细节。
